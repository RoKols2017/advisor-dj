## 🏗️ Архитектура

### Структура проекта
Проект построен на Django 5.2 с архитектурой MVT (Model-View-Template).

**Основные компоненты:**
- **accounts/** - управление пользователями (кастомная модель User с интеграцией AD)
- **printing/** - основная бизнес-логика (события печати, принтеры, статистика)
- **config/** - настройки проекта (разделены по окружениям)
- **printing/print_events_watcher.py** - демон для автоматического импорта файлов

**Модели данных:**
- `User` (расширенная AbstractUser) - пользователи системы
- `Department`, `Building` - организационная структура
- `Printer`, `PrinterModel` - принтеры и их модели
- `PrintEvent` - события печати (основная сущность)
- `Computer`, `Port` - вспомогательные сущности

**Архитектурные решения:**
- ✅ Разделение settings по окружениям (base/development/production/docker/test)
- ✅ Бизнес-логика вынесена в сервисы (`printing/services.py`, `accounts/services.py`)
- ✅ Использование signals для инвалидации кэша
- ✅ Docker-композиция для развертывания (web + watcher + db)

**Проблемы архитектуры:**
- ❌ Нарушение DRY: дублирование логики импорта (см. `printing/forms.py` vs `printing/services.py`)
- ❌ Неясная ответственность: `printing/importers.py` - тонкая обертка без добавленной стоимости
- ❌ Отсутствие слоя валидации данных (валидация размазана по views и services)
- ❌ Смешанная ответственность: `PrintTreeView` содержит сложную бизнес-логику построения дерева

## 🚨 Критические проблемы

### 1. **БАГ: Отсутствует параметр `model` при создании PrinterModel**

**Файл:** `printing/services.py:61`

```python
return PrinterModel.objects.create(code=normalized, manufacturer=manufacturer)
```

**Проблема:** В модели `PrinterModel` есть поля `code`, `manufacturer`, `model`, но при создании передается только `code` и `manufacturer`. Поле `model` остается пустым, что нарушает целостность данных.

**Последствия:**
- Неполные данные в БД
- Проблемы при отображении `__str__` модели (использует `manufacturer` и `model`)
- Возможные ошибки в UI/логах

**Исправление:**
```python
# Необходимо парсить model_code более умно
parts = normalized.split(maxsplit=1)
manufacturer = parts[0] if parts else normalized
model = parts[1] if len(parts) > 1 else ""
return PrinterModel.objects.create(
    code=normalized, 
    manufacturer=manufacturer,
    model=model
)
```

### 2. ✅ **ИСПРАВЛЕНО: Игнорирование фильтров дат в PrintTreeView**

**Файл:** `printing/views.py:72-117`

**Бывшая проблема:** Вьюха фильтровала query по `start_date` и `end_date`, но затем вызывала сервис с `None` вместо отфильтрованных значений. Это означало, что фильтры дат не работали для дерева печати.

**Последствия (устранены):**
- ❌ Пользователь указывал даты, но получал данные за все время
- ❌ Неправильная статистика
- ❌ Проблемы с производительностью на больших объемах данных

**Решение (2024-12-27):**
1. ✅ Добавлены даты по умолчанию: с первого числа текущего месяца до сегодня (для всех страниц с фильтрами)
2. ✅ Исправлена передача дат в `get_statistics_data()` - теперь всегда передаются реальные значения
3. ✅ Добавлена обязательная фильтрация по датам в `services.py` для предотвращения загрузки всех данных
4. ✅ Улучшено кэширование: для прошлых периодов кэш хранится 24 часа вместо 5 минут
5. ✅ Добавлены фильтры по датам на страницу "Статистика" с поддержкой фильтрации данных
6. ✅ Добавлены даты по умолчанию на страницу "События печати" (PrintEventsView)
7. ✅ Исправлено отображение отделов: показывается только название без кода на всех страницах (Дерево событий, Статистика, фильтры Событий печати)
8. ✅ Фильтрация ненулевых значений: на странице статистики показываются только отделы и пользователи с данными

**Текущая реализация:**
- При открытии страниц ("Дерево событий", "Статистика", "События печати") без параметров автоматически устанавливается период: первое число текущего месяца - сегодня
- Фильтрация по датам работает корректно на всех страницах
- Производительность значительно улучшена за счет ограничения периода по умолчанию
- В UI используются только названия отделов (без кода)

### 3. **БАГ: Дублирование транзакций в import_print_events**

**Файл:** `printing/services.py:141`

```python
for event in events:
    try:
        with transaction.atomic():
            # ... обработка события ...
```

**Проблема:** Каждое событие обрабатывается в отдельной транзакции. При большом объеме данных это приводит к:
- Множеству коммитов в БД (медленно)
- Невозможности откатить весь импорт при ошибке
- Потенциальным race condition при параллельном импорте

**Рекомендация:** Обернуть весь цикл в одну транзакцию (если не нужна частичная обработка) или использовать батчинг.

---

### 4. **БАГ: Отсутствие индексов на важных полях**

**Файл:** `printing/models/print_event.py`

**Проблема:** Отсутствуют индексы на:
- `user_id` (FK) - используется в фильтрах и агрегациях
- `printer_id` (FK) - используется в фильтрах
- Составной индекс `(user_id, timestamp)` для быстрых выборок по пользователю за период
- Составной индекс `(printer_id, timestamp)` для статистики по принтеру

**Последствия:**
- Медленные запросы при фильтрации по пользователю/принтеру
- Проблемы с производительностью на больших объемах данных

**Текущие индексы:**
```python
indexes = [
    models.Index(fields=['document_id']),  # ✅ есть
    models.Index(fields=['job_id']),       # ✅ есть
    models.Index(fields=['timestamp']),    # ✅ есть
]
# ❌ НЕТ индексов на user_id, printer_id, составных
```

### 5. **БАГ: Небезопасная обработка исключений**

**Файл:** `printing/services.py:213`

```python
except Exception as e:  # noqa: BLE001
    msg = f"Event import error: {e}"
    logger.error(msg)
    errors.append(msg)
```

**Проблема:** Ловятся ВСЕ исключения, включая системные (`KeyboardInterrupt`, `SystemExit`), что может привести к неожиданному поведению.

**Рекомендация:** Ловить только конкретные исключения или использовать `BaseException` только для критических случаев.

## 🔍 Логические ошибки

### 6. **Отсутствие валидации уникальности job_id в БД**

**Файл:** `printing/models/print_event.py`

**Проблема:** Поле `job_id` используется для дедупликации (строка 153 в services.py), но в модели нет `unique=True`. Это означает, что возможны дубликаты при параллельном импорте.

**Последствия:**
- Race condition: два процесса могут одновременно проверить `exists()` и создать дубликат
- Нарушение целостности данных

**Исправление:**
```python
job_id = models.CharField('ID задания', max_length=64, db_index=True, unique=True)
```

---

### 7. **Некорректная обработка отсутствующих пользователей**

**Файл:** `printing/services.py:180-184`

```python
try:
    user = User.objects.get(username__iexact=username)
except User.DoesNotExist:
    errors.append(f"User not found: {username}")
    continue
```

**Проблема:** Если пользователь не найден, событие просто пропускается. Это может привести к потере данных, если пользователи импортируются позже событий.

**Рекомендации:**
1. Автоматически создавать пользователя (если это допустимо бизнес-логикой)
2. Или сохранять "сырые" события в отдельную таблицу для последующей обработки
3. Или как минимум логировать критическое предупреждение

---

### 8. **Проблема с кэшированием в PrintTreeView**

**Файл:** `printing/views.py:98-101`

**Проблема:** Ключ кэша формируется из `start_date` и `end_date` из запроса, но данные всегда берутся за все время (`start_date=None, end_date=None`). Это означает:
- Кэш работает некорректно (разные ключи для одинаковых данных)
- Фильтры не применяются (см. баг #2)

---

### 9. **Дублирование логики в forms.py и services.py**

**Файл:** `printing/forms.py:48-87` vs `printing/services.py:100-133`

**Проблема:** Логика импорта пользователей дублируется:
- `UserImportForm.process_file()` - использует `User.objects.update_or_create()` напрямую
- `import_users_from_csv_stream()` - та же логика, но через сервис

**Последствия:**
- Нарушение DRY
- Разная обработка ошибок
- Сложность поддержки (изменения нужно делать в двух местах)

**Рекомендация:** Удалить `process_file()` из формы и использовать сервис.

## ⚡ Проблемы производительности

### 10. **N+1 запросы в get_statistics_data**

**Файл:** `printing/services.py:250-258`

```python
department_stats = (
    Department.objects.annotate(
        total_pages=Sum("users__print_events__pages"),
        total_documents=Count("users__print_events"),
        total_size=Sum("users__print_events__byte_size"),
    )
    .order_by("-total_pages")
    .all()
)
```

**Проблема:** При обращении к `users__print_events` Django выполняет множественные JOIN'ы. На больших объемах данных это может быть медленно.

**Рекомендация:** Использовать `prefetch_related` для оптимизации или переписать запрос с использованием `values()` и агрегации.

---

### 11. **Отсутствие пагинации в StatisticsView**

**Файл:** `printing/views.py:54-65`

**Проблема:** В `StatisticsView` выводятся все отделы и топ-10 пользователей без возможности пагинации. При большом количестве отделов это может привести к проблемам с производительностью и UX.

---

### 12. **Неэффективное построение дерева в PrintTreeView**

**Файл:** `printing/views.py:104-166`

**Проблема:** Дерево строится в Python-коде после получения всех данных из БД. Это:
- Загружает все данные в память
- Выполняет сложную обработку на уровне приложения
- Может быть медленным на больших объемах

**Рекомендация:** Рассмотреть использование рекурсивных CTE в PostgreSQL или денормализацию данных.

## 🔒 Проблемы безопасности

### 13. **CSRF exempt на ImportPrintEventsView**

**Файл:** `printing/views.py:198`

```python
@method_decorator(csrf_exempt, name='dispatch')
class ImportPrintEventsView(LoginRequiredMixin, View):
```

**Проблема:** Отключена CSRF-защита для эндпоинта импорта. Это позволяет злоумышленнику выполнить CSRF-атаку, если пользователь авторизован.

**Рекомендация:** 
1. Убрать `csrf_exempt` и использовать стандартную CSRF-защиту Django
2. Или добавить токен авторизации (например, через `IMPORT_TOKEN` из ENV)

---

### 14. **Отсутствие валидации размера файла при импорте**

**Файл:** `printing/views.py:180-195`

**Проблема:** Нет ограничения на размер загружаемого файла. Злоумышленник может загрузить огромный файл, что приведет к:
- Исчерпанию памяти
- DoS-атаке
- Длительной блокировке процесса

**Рекомендация:** Добавить проверку размера файла перед обработкой.

---

### 15. **Слабая валидация входных данных в импорте**

**Файл:** `printing/services.py:136-217`

**Проблемы:**
- Нет валидации типов данных перед парсингом
- Нет ограничений на длину строк
- Нет проверки на SQL-инъекции (хотя ORM защищает, но лучше валидировать)

**Пример уязвимости:**
```python
document_id = int(event.get("Param1") or 0)  # Может быть огромным числом
document_name = event.get("Param2", "")  # Может быть очень длинной строкой
```

### 24. **Кэширование QuerySet'ов**

**Файл:** `printing/services.py:259, 274`


**Вопрос:** В одном случае кэшируется QuerySet, в другом - список. Это неконсистентно. QuerySet'ы могут быть "устаревшими" при кэшировании (связи с БД могут разорваться). Почему разный подход?
