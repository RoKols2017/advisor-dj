# 📊 Анализ проекта Print Advisor

## 🏗️ Архитектура проекта

### Структура приложений

#### 1. **accounts/** — Управление пользователями
- Кастомная модель `User` (расширенная AbstractUser)
- Интеграция с Active Directory (Windows-аутентификация)
- Связь с отделами (Department)
- Сервисы аутентификации (`backends.py`, `services.py`)

#### 2. **printing/** — Основная бизнес-логика
- **Модели:**
  - `PrintEvent` — события печати (основная сущность)
  - `Printer`, `PrinterModel` — принтеры и модели
  - `Department`, `Building` — организационная структура
  - `Computer`, `Port` — вспомогательные сущности
- **Views:** Dashboard, статистика, импорт, фильтрация
- **Services:** Импорт данных, статистика, агрегации
- **Watcher:** Автоматический мониторинг каталога для импорта файлов

#### 3. **config/** — Настройки проекта
- Разделение settings по окружениям:
  - `base.py` — базовые настройки
  - `development.py` — разработка
  - `production.py` — продакшен
  - `docker.py` — Docker окружение
  - `test.py` — тестирование
- Логирование (`logging.py`)
- URL-маршрутизация (`urls.py`)

### Технологический стек

**Backend:**
- Python 3.13
- Django 5.2.1
- PostgreSQL 15 (production) / SQLite (development)
- Gunicorn (WSGI сервер)

**Frontend:**
- Django Templates
- Bootstrap 5
- django-tables2 (таблицы)
- django-filter (фильтрация)
- django-import-export (импорт/экспорт)

**DevOps:**
- Docker & Docker Compose
- Nginx (reverse proxy)
- Watchdog (мониторинг файлов)
- Systemd (автозапуск)

**Качество кода:**
- ruff (линтер)
- black (форматирование)
- mypy (типизация, strict mode)
- pytest (тестирование)
- pytest-cov (покрытие)

## 📈 Функциональность

### Основные возможности

1. **Мониторинг печати**
   - Отслеживание всех событий печати
   - Детальная информация о заданиях
   - Фильтрация по датам, пользователям, принтерам, отделам

2. **Статистика и аналитика**
   - Dashboard с общей статистикой за 30 дней
   - Статистика по отделам и пользователям
   - Иерархическое дерево печати (отдел → принтер → пользователь → документ)
   - Агрегация данных (страницы, документы, размер)

3. **Импорт данных**
   - Импорт пользователей из CSV (Active Directory)
   - Импорт событий печати из JSON
   - Автоматический импорт через watcher (мониторинг каталога)
   - Дедупликация по `job_id`

4. **Управление пользователями**
   - Кастомная модель User с ФИО и отделами
   - Интеграция с Windows-аутентификацией
   - Автоматическое создание пользователей при импорте событий

## 🐳 Docker-архитектура

### Сервисы

1. **nginx** (reverse proxy)
   - Единая точка входа (порт 80/443)
   - Маршрутизация запросов к web-сервису
   - SSL/TLS терминация

2. **web** (Django приложение)
   - Gunicorn с 3 воркерами
   - Health check endpoint (`/health/`)
   - Статические файлы через WhiteNoise
   - Логирование в файлы и консоль

3. **watcher** (демон мониторинга)
   - Отслеживание каталога `data/watch/`
   - Автоматическая обработка JSON/CSV файлов
   - Перемещение в `processed/` или `quarantine/`

4. **db** (PostgreSQL 15)
   - Персистентное хранилище (volume `pgdata`)
   - Health checks
   - Автоматический перезапуск

### Сети Docker

- `advisor-network` — изолированная сеть для сервисов приложения
- `reverse-proxy-network` — общая сеть для Nginx и backend

## ⚠️ Проблемы и риски

### Критические проблемы (исправлены частично)

1. ✅ **Фильтрация дат в PrintTreeView** — ИСПРАВЛЕНО
   - Было: фильтры не применялись
   - Стало: добавлены даты по умолчанию, фильтрация работает

2. ✅ **Создание PrinterModel** — ИСПРАВЛЕНО
   - Было: поле `model` не заполнялось
   - Стало: парсинг `model_code` с заполнением всех полей

3. ✅ **Уникальность job_id** — ИСПРАВЛЕНО
   - Было: отсутствовал `unique=True`
   - Стало: добавлен `unique=True` в модели

4. ✅ **Индексы** — ЧАСТИЧНО ИСПРАВЛЕНО
   - Добавлены индексы на `timestamp`, `job_id`, `document_id`
   - Добавлены составные индексы `(user, timestamp)`, `(printer, timestamp)`

### Средние проблемы

1. **Обработка отсутствующих пользователей**
   - Текущее: автоматическое создание пользователя при импорте событий
   - Риск: пользователи создаются без ФИО и отдела
   - Рекомендация: логировать предупреждения, обновлять при импорте CSV

2. **Валидация размера файла**
   - ✅ Добавлены ограничения: 10 МБ для CSV, 50 МБ для JSON
   - ⚠️ Нет ограничения на размер тела запроса (для JSON через POST)

3. **CSRF защита для импорта**
   - Текущее: `csrf_exempt` для `ImportPrintEventsView`
   - Риск: CSRF-атаки при авторизованном пользователе
   - Рекомендация: использовать токен авторизации (`IMPORT_TOKEN`)

### Низкие проблемы (улучшения)

1. **Дублирование логики**
   - Логика импорта частично дублируется в `forms.py` и `services.py`
   - Рекомендация: использовать только сервисы

2. **N+1 запросы**
   - В некоторых местах используются `select_related` / `prefetch_related`
   - Рекомендация: проверить все запросы на N+1

3. **Неиспользуемый код**
   - Модель `OrganizationalUnit` не используется
   - Модуль `printing/importers.py` — тонкие обертки
   - Рекомендация: удалить или задокументировать

4. **Документация архитектуры**
   - Файл `docs/concepts/architecture-notes.md` не заполнен
   - Рекомендация: добавить описание архитектуры
